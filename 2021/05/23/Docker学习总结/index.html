Docker学习总结

比较 Docker 和传统虚拟化方式的不同之处：

  • 传统虚拟机技术是虚拟出一套硬件后，在其上运行一个完整操作系统，在该系统上再运行所需应用进程；
  • docker容器内的应用进程直接运行于宿主的内核，容器内没有自己的内核，而且也没有进行硬件虚拟，因此容器要比传统虚拟机更为轻便。
  • 每个容器之间互相隔离，每个容器有自己的文件系统 ，容器之间进程不会相互影响，能区分计算资源。

Docker容器的好处：一次构建，随处运行

  • 更快速的应用交付和部署：
        传统的应用开发完成后，需要提供一堆安装程序和配置说明文档，安装部署后需根据配置文档进行繁杂的配置才能正常运行。Docker化之后只需要交付少量容器镜像文件，在正式生产环境加载镜像并运行即可，应用安装配置在镜像里已经内置好，大大节省部署配置和测试验证时间。
  • 更便捷的升级和扩缩容：
        随着微服务架构和Docker的发展，大量的应用会通过微服务方式架构，应用的开发构建将变成搭乐高积木一样，每个Docker容器将变成一块“积木”，应用的升级将变得非常容易。当现有的容器不足以支撑业务处理时，可通过镜像运行新的容器进行快速扩容，使应用系统的扩容从原先的天级变成分钟级甚至秒级。
  • 更简单的系统运维：
        应用容器化运行后，生产环境运行的应用可与开发、测试环境的应用高度一致，容器会将应用程序相关的环境和状态完全封装起来，不会因为底层基础架构和操作系统的不一致性给应用带来影响，产生新的BUG。当出现程序异常时，也可以通过测试环境的相同容器进行快速定位和修复。
  • 更高效的计算资源利用：
        Docker是内核级虚拟化，其不像传统的虚拟化技术一样需要额外的Hypervisor支持，所以在一台物理机上可以运行很多个容器实例，可大大提升物理服务器的CPU和内存的利用率。

Docker使用心得：

  • 首先，我们在docker中进行容器的配置，也就是搭建“集装箱”，我们以一个普通的JavaWeb项目为例
  • 项目需要：centos7，tomcat，mysql，redis，就以这四项为例
  • 比方说我们现在已经将项目的代码写好了，准备部署，或者交给运营来部署，原来就是打包成war包交给运营，运营在他的电脑上运行，但是容易出现系统版本不兼容，软件版本不兼容等众多问题，此时docker的好处就来了
  • 此时我们在docker中，先拉取centos7，tomcat，mysql，redis的镜像，每个当中都生成一个容器，将项目当中的tomcat，mysql，redis的所有信息都转成和容器匹配的，或者在项目的开始的时候就直接使用docker的容器，更方便，此时的各个容器都是和项目相关的容器，都是根据项目进行配置的，包括数据，包等所有信息
  • 此时再docker容器上能够跑通项目之后，我们将每个容器通过commit生成镜像，然后可以推送到远端仓库，这样运维就可以在他那边这些镜像拉取下来，拉取下来的镜像就是我们的项目的所有信息，运维只需要将镜像生成容器来运行即可，这时候就不存在所谓的不兼容等问题
  • 每个容器都相当于一个微型的linux系统，因此使用各个容器就和linux差不多

Docker的流程：

  • 使用dockerfile，然后build，编译成镜像，相当于自己写一个镜像的源码，自己编译到本地镜像库中
  • 或者直接从远端仓库拉取到本地镜像库中，docker pull
  • 此时已经拉取镜像到了本地镜像库，然后docker run生成并运行容器
  • 更新好的容器也可以再生成镜像保存到本地库中，相当于原来镜像的版本更新，docker commit
  • 此时已经提交到了本地镜像仓库
  • 当然也可以提交到阿里云或者公司库等远端库，具体需要看远端镜像库的操作（不同远端可能不同）
  • 此时运维或者其他人可以使用，拉取到他们的机器上直接运行即可，不用顾及兼容性等问题

Docker镜像原理：

  • 对于一个镜像而言，其底层有多层的文件系统，拉取一个镜像，实则镜像里面的众多文件系统都需要被拉取下来，一层套一层，一层一层的拉取，想“花卷”一样的结构
  • 官方解释：
        docker的镜像实际上由一层一层的文件系统组成，这种层级的文件系统UnionFS。
    bootfs(boot file system)主要包含bootloader和kernel, bootloader主要是引导加载kernel, Linux刚启动时会加载bootfs文件系统，在Docker镜像的最底层是bootfs。这一层与我们典型的Linux/Unix系统是一样的，包含boot加载器和内核。当boot加载完成之后整个内核就都在内存中了，此时内存的使用权已由bootfs转交给内核，此时系统也会卸载bootfs。
    rootfs (root file system) ，在bootfs之上。包含的就是典型 Linux 系统中的 /dev, /proc, /bin, /etc 等标准目录和文件。rootfs就是各种不同的操作系统发行版，比如Ubuntu，Centos等等。
  • 经典问题：平时我们安装进虚拟机的CentOS都是好几个G，为什么docker这里才200M？
        对于一个精简的OS，rootfs可以很小，只需要包括最基本的命令、工具和程序库就可以了，因为底层直接用Host的kernel，自己只需要提供 rootfs 就行了。由此可见对于不同的linux发行版, bootfs基本是一致的, rootfs会有差别, 因此不同的发行版可以公用bootfs。
  • 问题：为什么 Docker 镜像要采用这种分层结构？
         最大的一个好处就是 - 共享资源
     比如：有多个镜像都从相同的 base 镜像构建而来，那么宿主机只需在磁盘上保存一份base镜像，同时内存中也只需加载一份 base 镜像，就可以为所有容器服务了。而且镜像的每一层都可以被共享。
  • 特点：Docker镜像都是只读的，当容器启动时，一个新的可写层被加载到镜像的顶部。这一层通常被称作“容器层”，“容器层”之下的都都叫“镜像层”。

容器数据卷：

  • 使用数据卷的目的：Docker容器产生的数据，如果不通过docker commit生成新的镜像，使得数据做为镜像的一部分保存下来，
    那么当容器删除后，数据自然也就没有了。
  • 卷的设计目的就是数据的持久化，完全独立于容器的生存周期，因此Docker不会在容器删除时删除其挂载的数据卷
  • 数据卷其实就是相当于一个U盘的功能，开启了数据卷，宿主机和容器之间就开启了一个通道，容器当中的一个文件夹和宿主机中的一个文件夹形成映射关系，共享内容，此时容器中的内容就被同步到宿主机中，即使容器销毁，那么数据也被持久化到宿主机上
  • 数据卷可读可写，也就是是双方都在自己对应的文件夹中写内容，会同步到双方的文件夹中读取，当然也可以设置成只读模式，也就是容器只能读宿主机，不能在这个文件夹修改或者新建内容
  • 当容器重启的话，此时的容器不是关闭，而是重启，那么容器就会从数据卷中也就是从宿主机中加载回之前的数据
  • 数据卷容器：命名的容器挂载数据卷，其它容器通过挂载这个(父容器)实现数据共享，挂载数据卷的容器，称之为数据卷容器
  • 容器之间配置信息的传递，数据卷的生命周期一直持续到没有容器使用它为止，只要存在过继承关系，就可以访问容器数据卷

Dockerfile：

  • 什么是dockerfile：Dockerfile是用来构建Docker镜像的构建文件，是由一系列命令和参数构成的脚本（不同于shell脚本，他的一个命令会生成一个容器，修改提交生成镜像，再删除这个临时的容器）。相当于一个镜像的底层源码

  • 注意，每条保留字指令执行完，都会创建一个新的镜像层，并对镜像进行提交，这也是前面所说的“花卷”的结构
