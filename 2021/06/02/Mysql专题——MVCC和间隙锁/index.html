Mysql专题——MVCC和间隙锁

相关概念

什么是MVCC（背景）？

多个事务操作同一条数据的时候，即使这些操作既包括读，也包括写，此时的过程就是数据库事务并发的过程，在InnoDB中，为了提高刚刚所说的读写并发能力，处理好并发情况下的读写冲突，读取数据不一致的问题，这里采用了一种叫做多版本并发控制（MVCC）的方法，好处是能够做到不加锁，能够非阻塞的并发读。关于数据的读取，还会分为两种读取方式，分别为当前读和快照读

数据库的并发场景？

读-读：

  • 不存在任何问题，也不需要并发控制

读-写：

  • 有线程安全问题，可能会造成事务隔离性问题，可能遇到脏读，幻读，不可重复读
  • 也就是一个事务读的过程中，其他事务进行了相关写操作，造成了读取数据不一致

写-写：

  • 有线程安全问题，多个事务同时写，可能会存在更新丢失或者也叫数据覆盖问题

什么是当前读？什么是快照读？

当前读：

  • 什么是当前读：读取到的内容一定要是数据的最新版本，保证的数据的准确性，其他的并发事务不能够并发的改变正在读取的数据
  • 本质：实际上，当前读是一种加锁的读，会对读取的记录进行加锁处理，保证当前读的准确性，是悲观锁的实现
  • 最常见的当前读：在进行增删改操作语句的时候的步骤是先读取数据再进行操作，此时的读就是当前读，此时的读被加了锁

快照读：

  • 什么是快照读：对比当前读，快照读是不进行加锁的读取，也就是说一个事务在读取一条数据的时候允许其他事务对读取的数据进行相关操作，是一个事务并发的过程，换句话说，快照读出现在隔离级别不是串行化的事务当中，否则就变成了当前读

  • 会出现什么问题：此时在无锁的情况下，就会出现多个事务读写数据不一致的情况，比如脏读，不可重复读，幻读等

  • 如何无锁解决该问题：对于数据的读取，由于受到了其他事务的操作，此时该条数据会产生多个版本，MVCC的作用是从这么多个版本当中按照一定的规则去挑选出一个版本，作为读取的结果，至于什么样的规则，在可重复读与读已提交的事务隔离级别中的规则是不一样的，后续会分类讲解

MVCC的优点或作用：

  • 在并发读写数据库时，可以做到在读操作时不用阻塞写操作，写操作也不用阻塞读操作，提高了数据库并发读写的性能
  • 同时还可以解决脏读，幻读，不可重复读等事务隔离问题，但不能解决更新丢失问题，因为丢失修改属于写-写冲突
  • 此时需要注意：在读已提交的隔离级别中，MVCC的规则解决了脏独的问题，在可重复读的隔离级别中，MVCC的规则解决了不可重复读以及快照读的幻读问题，也就是说对于幻读，只解决了一部分幻读问题，在MVCC原理当中，将分别讲解这三种情况的原理

MVCC原理分析

补充知识：

事务ID：

  • 每开启一个事务，这个事务都有属于自己的编号，叫做事务ID

隐式字段：

  • 查询过程中，查询出来的结果除了表中的字段之外，还包括三个隐藏的字段，是mysql自己提供的，如下：
  • DB_TRX_ID：操作该数据的事务ID
  • DB_ROLL_PTR：回滚指针，里面是一个地址，指向的是这条数据的上一个版本，或者叫做被修改之前的版本
  • DB_ROW_ID：自增ID，当数据库表没有提供自增主键索引或者唯一索引的时候数据库会自动分配的主键索引
  • flag删除字段（特殊）：该字段是真正的隐藏起来的，当删除数据的时候采用的是逻辑删除，将flag置为1，表示删除了数据

undo log（日志）：

  • 什么时候出现：只要开启事务，对数据操作，就会产生版本日志

  • 内容1：在这个日志中，会记录查询记录的所有版本，版本与版本之间通过回滚指针相连，形成了版本链，MVCC的核心原理就是从这个版本链当中从上到下按照规则的找，哪个符合规则，就将哪个版本读取出来

  • 内容2：每个版本的具体内容包括操作的数据以及上述隐藏字段

Read View（读视图）：

  • 什么是Read View：前文提到的按照某种规则来进行查找版本链，而这个Read View就是提供规则的标准，换句话说，规则就是以它为基准来指定，因此在RC和RR两种隔离级别中，Read View的生成方式不一样，自然也就导致隔离级别不一样

  • 什么时候产生Read View：由于MVCC基于的是快照读，产生的时间点根据隔离级别不同，时间也就不同：

    • RC：只要在事务中读一次，那么就重新生成一个Read View
    • RR：当读事务开启的时候，就会产生唯一一个Read View，只要这个读事务不结束，无论读多少次，Read View也不变

  • Read View的产生时机决定了RC和RR解决的事务并发问题的不同，也就是决定了这两种隔离级别

  • Read View的组成：

    • 当前版本链中，所有未提交事务的事务id数组
    • 读事务开启，第一次读的时候的活跃的（版本链中存在的）最大事务id（有的博主也说是再生成事务时候的事务ID，其实差不多）
    • 主要就是以上两部分（还有一个就是当前读事务的事务id，该部分不需要过度关注，因为我们知道在当前的读事务的操作中，无论做什么操作都是能读出来的，毕竟是在当前事务）

  • 读取规则：从开始读的时刻的版本链向下一个一个找，把走到的事务成为当前事务：

    • 如果当前事务id<未提交事务id数组中的最小值，说明当前事务的id已经提交了，那么此事务可读，可以直接读出来，不用继续向下找了，否则继续按照版本链向下
    • 如果当前事务id在未提交事务id数组当中，说明该事务未提交，则不可读（只有当前事务可读）
    • 如果当前事务id不在未提交事务id数组当中并且小于等于最大事务id，说明已经提交了，可读
    • 如果当前事务id大于了最大事务id，说明是在我开启读事务之后再进行的事务，此时不可读（也突出了快照的特点）

上述过程的相关结论：

  • 通过RC的Read View读取的结果发现，此时发生了不可重复读，但是完成了读已提交
  • 通过RR的Read View读取的结果发现，此时完成了可重复读

提出问题：

按照上面的RR的方式，对于其他事务的操作，肯定也包括插入，那么插入的时候由于Read View不变，那么插入数据的事务应该也不起作用，此时就不应该发生幻读，那么为什么说可重复读的隔离级别还是不能避免幻读呢？看如下演示情况：

幻读演示：包含快照读和当前读:

  • 假设表中有如下数据：

    id name
    1 jack

  • 开启如下事务，首先演示快照读

事务1 事务2
select *
insert 2 Nancy
commit
select *
  • 根据RR的MVCC可知，事务1的Read View不会变，所以事务2做的改变并不会被查询到，此时解决了快照读的幻读
  • 开启如下事务，首先演示当前读
事务1 事务2
select *
insert 2 Nancy
commit
insert 2 Linda
  • 此时的执行的事务1执行的insert报错，报错的内容是主键2已经存在了，说明在执行insert的时候首先执行了当前读，查到了表中已经多了一条数据，所以不能够再插入了，此时在事务1中的两次读取过程中，读到了多一条数据，此时发生了幻读

如何解决幻读？

  • 很明显可重复读的隔离级别没有办法彻底的解决幻读的问题，如果我们的项目中需要解决幻读的话也有两个办法：
  • 使用串行化读的隔离级别
  • MVCC + 间隙锁

间隙锁

相关概念

什么是间隙：

  • 在查询过程中，设置了索引，会按照索引进行查询，索引之间的范围或者第一个索引前面或者最后一个索引后面的范围都是间隙

什么是间隙锁：

  • 间隙锁（Gap Lock）是Innodb在RR隔离级别提交下为了解决幻读问题时引入的锁机制
  • 间隙锁通常是用来解决幻读的，我们都默认已经解决了不可重复的问题，因为MVCC，所以我们当前处在的是RR的隔离级别
  • 在RC的隔离级别不用确保多次读的一致性，因此也不存在幻读的概念
  • 我们通常研究的索引为唯一索引和普通索引
  • 下面通过示例来理解一下什么是间隙，以及如何加间隙锁，普通索引和唯一索引分别都是怎么加入间隙锁的

示例演示过程

索引为唯一索引：

id（主键索引） number
1 1
2 3
13 3
23 3
31 11
40 40

  • 此时的间隙为（无穷小，1），（1,2），（2,13），（13,23），（23,31），（31,40），（40，无穷大）

  • 命中唯一索引记录：

    此时只会对id=2，number=3这条记录加上行锁，不会产生间隙锁，因此，插入id=3，id=4等后面间隙当中都不会阻塞

  • 未命中索引记录：

    此时，没有查到id=3的记录，就叫做未命中唯一索引，此时产生间隙锁，id=3处在（2,13）这个间隙当中，如果有事务想要在这个间隙当中插入以id作为索引的数据，那么就会阻塞，比如insert into test(id, number) value(5, 3)，此时阻塞，因为5在2到13的间隙中

  • 范围内命中索引，此时升级为临建锁（行锁+间隙锁）：

    此时，在5到15这个范围，命中了id=13这条记录，此时命中了索引，索引id=13，number=3这条记录被加上了行锁，与此同时，5到15这个范围又处在了（2,13）和（13，23）这两个间隙当中，因此这两个间隙被加上了间隙锁，此时在两个范围内就不能够插入数据，因此也叫做临键锁，即既生成了行锁，又生成了间隙锁

  • 范围内未命中索引记录：

    此时6到10这个范围未命中索引，但是处在了（2,13）这个范围内，因此这个间隙被加上了间隙锁

索引为普通索引：

id（主键索引） number（普通索引）
1 1
2 3
13 3
23 3
31 11
40 40

  • 上述查询语句是按照number这个普通索引来进行查询的，那么对于number索引可以产生多个间隙：(无穷小,1)，(1,3)，(3,11)，(11，40)，(40，无穷大)

总结间隙锁的加锁规则：

唯一索引（主键索引）

  • 查询一条记录：

    • 命中索引加行锁
    • 未命中索引，在所属间隙加上间隙锁

  • 范围查询：

    • 命中索引，加临键锁（命中的索引记录加上行锁，其余处在间隙的部分加间隙锁）
    • 未命中索引记录，所处的间隙加上间隙锁

普通索引：

  • 对于普通索引列的查询，不管是是否命中索引，只要加锁，都会产生间隙锁（和唯一索引的区别）
  • 在普通索引跟唯一索引中，数据间隙的分析，数据行是优先根据普通索引排序，再根据唯一索引排序。
