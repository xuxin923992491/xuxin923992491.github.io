阶段性复习（Redis篇）

Created2021-06-10|Updated2021-06-10
|Post View:

概念

什么是redis？

redis就是一种nosql数据库，可以作为缓存使用

为什么使用redis缓存？

  • 高性能：用户第一次访问数据慢是因为要从磁盘上读取，将用户访问的数据存入缓存中，再次访问就可以从缓存中获取，所以速度很快，但是如果数据库中对应的数据改变之后，需要同步改变缓存中相应的数据即可
  • 高并发：直接操作缓存能够承受的请求远远大于直接访问数据库，所以考虑将部分数据转移到缓存中，这样获取一部分数据就不用经过数据库了

Redis模型

什么是redis的单线程模型？

Redis的单线程模型是指，在处理用户请求的过程中，使用了文件事件处理器的模型，这个模型是个单线程模型，具体包括：多个Socket，IO多路复用程序，文件事件分派器，事件处理器（含连接应答、命令请求、命令回复）

单线程模型的好处？

没有线程切换带来的开销，基于内存的读写操作效率高。

Redis的单线程模型流程：

采用IO多路复用机制，同时监听多个客户端的连接请求Socket，并且把所有产生事件的Socket压入一个队列中，由文件事件分派器有序地每次取出一个socket给对应的事件处理器进行处理（包括连接应答，请求命令，命令回复）

Redis和memcache区别：

数据类型方面：

  • redis支持更丰富的数据类型，不只是key-value类型，也提供list，set，zset，hash等数据结构的存储
  • 后者仅支持简单的数据类型

持久性：

  • 前者支持持久化，可将数据保存在磁盘中，重启的时候可以再次加载进行使用
  • 后者仅保存在内存中

集群：

  • 支持cluster模式
  • 后者没有原生的集群模式

线程模型：

  • 前者是单线程的多路IO复用模型
  • 后者是多线程，非阻塞IO复用的网络模型

Redis数据结构

String

  • 形式：简单的key-value
  • 应用：可以用来表示微博粉丝数等

Hash

  • 形式：适合存储对象，是field和value的映射表，适合存储对象，key表示某个对象，value里面是属性：值
  • 应用：一般应用在存储用户信息，商品信息等

List

  • 形式：redis实现的是双向链表，可以支持反向查找和遍历，更方便操作，但是带来了额外的内存开销，也可以基于list实现分页（lrange命令）
  • 应用：比如微博关注列表，粉丝列表，消息列表等

Set

  • 形式：和list类似，但是set可以自动排重，提供了某个成员是否在一个set集合内的重要接口，可以实现集合交并差的操作
  • 应用：实现共同关注、共同粉丝等功能

SortedSet

  • 形式：相比于set多了一个权重参数score，使集合元素能够按score进行有序排列
  • 应用：比如直播的礼物排行榜，直播间的热度等

Redis的淘汰机制

redis设置过期时间

目的：

为了保证实时性，需要redis设置一个过期时间，防止一直占用缓存，可以给一个expire time，就是过期时间，指定key可以存活的时间，那么设置一批key只存活一个小时，那么接下来的一小时后，redis是怎么对这批key进行删除的？

两种删除方式：

  • 定期删除：默认是每隔100s随机抽取设置了过期时间的key，检查是否过期，过期就删除掉，随机的原因是因为数据太多，全部遍历的话耗时耗性能

  • 惰性删除：定期删除导致了很多到期的没有删除，那么就还停留在内存当中，那么就要去主动去查一下到期的key了

  • 上述的两种删除仍然会导致内存中堆积大量过期key，导致redis会内存耗尽，那么就需要redis内存淘汰机制

内存淘汰机制

面向的问题

主要问题就是加入mysql有2000w数据，如何保证redis中的数据都是热点实时数据而不是长时间不用的旧数据

六种淘汰策略

  • volatile-lru：从已设置过期时间的数据块挑选最近最少使用的数据淘汰
  • volatile-ttl：从已设置过期时间的数据集中挑选将要过期的数据淘汰
  • volatile-random：从已设置过期时间的数据集中任意选择淘汰
  • allkeys-lru：内存不足以容纳新写入数据时，在键空间中移除最少使用的key（最常用）
  • allkeys-random:从数据集中任意选择淘汰
  • no-eviction：禁止驱逐数据，也就是说当内存不足以容纳新写入数据时，新写入操作会报错

Redis持久化机制

目的

redis是内存数据库，它把数据存储在内存中，这样在加快读取速度的同时也对数据安全性产生了新的问题，即当redis所在服务器发生宕机后，redis数据库里的所有数据将会全部丢失。为了解决这个问题，redis提供了持久化功能——RDB和AOF（Append Only File）。

RDB

什么是RDB

  • RDB（Redis DataBase）是 Redis 默认的持久化方案。
  • 在指定的时间间隔内，执行指定次数的写操作，则会将内存中的数据快照写入到磁盘的dump.rdb文件。
  • Redis重启会通过加载dump.rdb文件来恢复数据。

异步进行持久化

  • 目的：不会阻塞主进程处理请求

  • Redis进行rdb持久化的时候，Redis会复制一个与当前进程一样的子进程，来异步进行持久化，主进程不需要进行任何IO操作

  • 因为redis在处理用户请求是个单线程模型，所以如果利用主线程去持久化，就会造成处理用户请求的阻塞，所以要fork一个子进程来异步完成持久化，这样就不会阻塞主进程处理请求。

持久化策略

  • 总述：Redis 在seconds秒内key改变changes次，Redis把快照内的数据保存到磁盘中一次。具体如下

  • 1分钟内改变了1万次

  • 5分钟内改变了10次

  • 15分钟内改变了1次

优点

  • 适合大数据量的恢复，相比AOF, rdb数据恢复速度快。
  • 通过子进程进行持久化，不会影响主进程处理用户请求。

缺点

  • 由于rdb持久化策略，每隔几分钟持久化到磁盘一次，所以如果在这几分钟内，redis宕机了，则这几分钟的数据就丢失了
  • 所以有数据不一致性问题。

AOF

什么AOF

  • AOF(Append Only File)，Redis 默认不开启。
  • 它的出现是为了弥补RDB的不足（数据的不一致性），所以它采用日志的形式来记录每个写操作，并追加到appendonly.aof文件中。
  • Redis 重启会根据日志文件将写指令从前到后执行一次以完成数据的恢复工作。

持久化策略

  • 异步持久化，具体如下
  • always：同步持久化，每次发生数据变化会立刻写入到磁盘中。性能较差但数据完整性比较好（慢，安全）
  • everysec：出厂默认推荐，每秒异步记录一次（默认值）
  • no：不即时同步，由操作系统决定何时同步。

AOF的重写机制

  • 目的：

    • AOF采用文件追加方式，文件会越来越大为避免出现此种情况，因此新增了重写机制
    • 当AOF文件的大小超过所设定的阈值时，Redis就会启动AOF文件的内容压缩，只保留可以恢复数据的最小指令集。

  • 过程：

    • AOF文件持续增长而过大时，会fork出一条新进程来将文件重写(也是先写临时文件最后再rename)，遍历新进程的内存中数据，每条记录有一条的Set语句。
    • 重写aof文件的操作，并没有读取旧的aof文件，而是将整个内存中的数据库内容用命令的方式重写了一个新的aof文件，这点和快照有点类似。
    • Redis会记录上次重写时的AOF大小，默认配置是当AOF文件大小是上次rewrite后大小的一倍且文件大于64M时触发。当然，也可以在配置文件中进行配置。

优点

把每一条写操作都记录到磁盘中，数据一致性和完整性较好。

缺点

所有的写指令都记录下来，数据恢复要把所有指令都执行一遍，恢复数据时效率不如rdb

Redis安全问题

缓存穿透

发生情景

  • 指查询一个一定不存在的数据，如果从数据库层查不到数据则不写入缓存
  • 这将导致这个不存在的数据每次请求都要到DB去查询，可能导致DB挂掉。
  • 总结：请求数据库也不存在的数据

解决方案

  • 有两种，如下：

  • 查询返回的数据为空，仍把这个空结果进行缓存，但过期时间会比较短

  • 布隆过滤器：将所有可能存在的数据哈希到一个足够大的bitmap中，一个一定不存在的数据会被这个bitmap拦截掉，从而避免了对DB的查询。

缓存击穿

发生情景

  • 对于设置了过期时间的key，缓存在某个时间点过期的时候，恰好这时间点对这个Key有大量的并发请求过来
  • 这些请求发现缓存过期一般都会从后端DB加载数据并回设到缓存，这个时候大并发的请求可能会瞬间把DB压垮。
  • 总结：刚好过期，还没从数据库加载回来，大量请求过来

解决方案

  • 使用互斥锁：当缓存失效时，不立即去加载数据库，先使用如Redis的setnx去设置一个互斥锁，并发线程拿到锁才能去访问数据库，并回设缓存。这样同一时刻只有一个线程可以拿到锁，然后去访问db,并且将访问到的数据回设缓存，不会导致大量并发请求同时落到db上。
  • 永远不过期：物理不过期，但逻辑过期（后台异步线程去刷新）。

缓存雪崩

发生情景

  • 设置缓存时采用了相同的过期时间，导致缓存在某一时刻同时失效，请求全部转发到DB，DB瞬时压力过重雪崩。
  • 与缓存击穿的区别：雪崩是很多key，击穿是某一个key缓存。

解决方案

  • 将缓存失效时间分散开，比如可以在原有的失效时间基础上增加一个随机值，比如1-5秒随机，这样每一个缓存的过期时间的重复率就会降低，就很难引发集体失效的事件。

  • 使用互斥锁，当缓存数据失效时，保证只有一个请求能够访问到数据库，并更新缓存，其他线程等待并重试；

  • 采用合适的淘汰策略，快过期的就淘汰掉，同时保证redis的高可用

  • 发生了之后，采取本地缓存+限流降级

    • 限流：5000个，只能走2000个
    • 降级：剩余3000赋默认值，或者友情提示或者空白，总比直接把数据库干蹦了好

  • 发生之后，利用redis持久化机制保存的数据尽快恢复缓存

Redis缓存不一致问题

Redis数据读写过程

  • 读数据：先读缓存，缓存有，直接返回，缓存没有，在从数据库读，并同步到缓存中。
  • 写数据：写数据，先删除缓存，然后更新数据库。

不一致发生原因

  • 上述读写操作中不是原子操作，删除与更新可能存在多线程问题，出现问题的过程如下：
  • 当写数据时，redis缓存被删除，然后再去更新数据库
  • 当数据库更新操作没有完成前（因为读的速度快于写的速度），又来了一条读指令，这时候缓存中没有数据，只能去数据库中读
  • 数据库中还没完成更新，所以读到的是旧数据，并把旧数据同步到了缓存中
  • 如果缓存没有设置过期时间，则缓存中的旧数据永远存在，以后再次读缓存永远都是旧数据，造成缓存和数据库数据不一致。
  • 总结：数据要变，变之前先读了，缓存被更新为旧值，数据库写完是新值

解决方案

  • 延时双删：每次更新数据库之前删除缓存，当更新完数据库之后再删除一次缓存，防止在没更新完数据库之前就因为读指令使缓存读到了数据库的旧数据。
  • 先更新数据库，再删除缓存：一条写指令先更新数据库，这时候即使有读指令读到了缓存的旧数据，当数据库更新完成后，会删除缓存的旧数据，再次读取数据的时候，会从数据库读新数据到缓存中。
  • 先更新数据库，再删除缓存，若删除缓存失败，可以通过增加消息列队来双重保证，过程如下：
    • 更新数据库数据
    • 缓存因为种种问题删除失败
    • 将需要删除的key发送至消息队列
    • 自己消费消息，获得需要删除的key
    • 继续重试删除操作，直到成功

Redis实现分布式锁

目的

当多个主机中的多个线程取访问共享数据的时候，为保证数据的安全性，需要使用分布式锁。

场景举例

当缓存击穿，或者缓存雪崩，防止同一时间大量的客户端请求（分布式的多线程）同时落到数据库，造成数据库宕机，需要设置分布式锁。

加锁过程

原理

  • 执行语句：SET my_key my_value NX PX milliseconds。
  • NX表示只有当键key不存在的时候才会设置key的值，PX表示设置键key的过期时间，单位是毫秒。
  • 加锁的过程就是设置key value的过程，释放锁的过程就是删除key value的过程。

过程

  • 加锁之前要通过NX判断Key是否存在，不存在说明没有其他线程占有该锁，存在说明有其他线程占有该锁，次操作不成功。
  • 设置过期时间的原因是保证锁的释放，比如某个线程加锁后，执行dosome()方法，还没释放锁之前就宕机了，这样导致所无法释放，别的线程也无法获取该锁。所以PX设置过期时间，即使该线程宕机无法释放锁，key也会到期自动删除。
  • SET my_key my_value NX PX milliseconds是原子性命令，也就是说在加锁过程中不会有其他线程获取锁的可能。

解锁过程

原理

释放锁的过程就是删除key的过程。

过程

  • 由于key设置了过期时间，如果在执行dosome（即执行任务）过程中, key就过期了，其他线程就可以获取到该锁，等dosome()执行完毕，通过删除key释放锁，可能释放的就是别的线程所占有的锁。
  • 为解决这个问题，每个线程加锁的时候set key value(随机值)，设置的Value增加随机值，即每个线程加锁的value不同，这样在执行释放锁的时候，要先判断value=get(key)是否等于之前设置好的value,如果相同说明释放的是本线程的锁，如果不相等说明已经有其他线程占有了该锁，不需要再释放锁了（删除key）
  • 总结，在A线程释放锁之前，可能key到期，B线程拿到了锁，A执行完任务，要释放锁，可能会释放B线程的锁，此时出现线程安全问题，为了保证原子操作，写成lua脚本保证原子性，保证A执行完任务之后，B才能拿到锁，此时没有了线程安全问题，既解决了删锁的问题，又解决了A,B同时执行造成数据不一致的问题。

Redis实现分布式锁特点

  • 有锁的失效机制（过期时间），防止死锁，防止无法释放锁
  • 非阻塞性，获取锁不用阻塞等待，获取不到即可返回
  • 基于内存操作，高性能。
  • 流程：
    • 加锁：设置key value
    • 执行代码：dosome（）
    • 解锁：删除key
