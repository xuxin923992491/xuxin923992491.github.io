Mycat理论

概念

为什么用

  • Java与数据库紧耦合。
  • 高访问量高并发对数据库的压力。
  • 读写请求数据不一致

作用：

读写分离

  • 数据经过mycat，可以将写命令发送给指定的mysql，读命令发给其他的服务器
  • 前提是mysql集群配置了主从复制
  • 总结：写指令交给主机，主机用来写数据，从机用来读数据，执行读指令，主机写完的数据，从机进行赋值

数据分片

  • 垂直拆分，即分库
  • 水平拆分，即分表
  • 垂直+水平拆分，即分库分表
  • 总结：建表插入数据，通过mycat，就能够自动让这个表被创建在哪个mysql当中，自动让数据插入到哪个mysql的哪个表中

多数据源整合

Mycat原理

  • 起到了“拦截”的作用
  • 拦截了用户发送过来的 SQL 语句，首先对 SQL 语句做了一些特定的分析：如分片分析、路由分析、读写分离分析、缓存分析等
  • 经过了上述分析，就知道数据应该操作哪个库或者哪个表，然后将此 SQL 发往后端的真实数据库
  • 将返回的结果做适当的处理，最终再返回给用户。
  • 这种方式把数据库的分布式从代码中解耦出来，程序员察觉不出来后台使用 Mycat 还是 MySQL

主从复制原理

mysql主从复制原理

  • 主机会把所有的写操作写到日志中，叫做binlog日志，是一个二进制日志
  • 从机不能够直接访问主机，需要去读取binlog日志
  • 此时日志记录了所有的主机的写操作
  • 将读取到的写操作拿给从机，注意不是马上让从机开始写操作
  • 需要写入进Relay log日志
  • 然后从机根据Relay log让从机读取执行

redis主从复制原理

  • Redis的从机找主机
  • 主机把所有的数据写入到一个持久化文件中（RDB）
  • 主机把持久化文件传给从机
  • 从机用主机的RDB文件替换自己的RDB文件
  • 从机把替换后的RDB文件读回到从机
  • 此时主机做任何工作，从机都会实时复制

和redis主从复制相同和区别

  • 相同点：主机只能有1台，从机可以有多台
  • 不同点：redis是从头开始复制，也就是直接就从主机开始复制;mysql从接入点开始复制，也就是从binlog的位置
  • 什么叫做从接入点开始复制：就是从配置了主从复制的那一刻开始复制，如果主机事先创建了一个表，那么就不会从机复制过来可能报错，而redis的主从复制就会把整个内容进行复制

mysql主从复制存在的问题

会发现mysql主从复制会进行两次io，分别在binlog和relay log，此时存在延时性问题

  • 登录进去后相当于通过mycat来操作mysql，进行类似于负载均衡的操作

Mycat实现读写分离

  • 什么是读写分离：就是让主机写，读只交给从机

  • 演示前提：在写主机插入：insert into mytbl values (1,@@hostname); ，此时主从不一致，因为主从的主机名不一样

  • 此时需要设置mycat配置文件中的scheme.xml文件，调整 < dataHost>里面的balance属性，按照下面四种方式取值：

    • balance=”0”, 不开启读写分离机制，所有读操作都发送到当前可用的 writeHost 上。
    • balance=”1”，全部的 readHost 与 stand by writeHost 参与 select 语句的负载均衡，简单的说，当双主双从 模式(M1->S1，M2->S2，并且 M1 与 M2 互为主备)，正常情况下，M2,S1,S2 都参与 select 语句的负载均衡。
    • balance=”2”，所有读操作都随机的在 writeHost、readhost 上分发。
    • balance=”3”，所有读请求随机的分发到 readhost 执行，writerHost 不负担读压力

  • 上述四种方式，对于双主双从的情况，一般balance设置为1或3，单主单从为了模拟切换读的效果，可以选择2，如果选择了0，那么就只有s1进行读，也就是读的结果只有从机的主机名

  • balance为2的话，查询出来的name字段就会在两个主机名进行来回切换

  • 设置完配置文件之后，启动mycat即可

对于双主双从

  • 同上面类似，只不过两个主机之间要互为主从，这样保证其中一个宕机另一个能够补上作为写主机
  • 通常balance=1，这样其中一个主机写的时候，另一个主机和从机的作用一样，作为读主机，当执行演示操作的时候，查询出来的name字段会在M2，S1,S2这三个主机名来回切换的
  • 双主双从保证了数据库读写分离的高可用性
  • 双主双从的抗风险能力高，当M1宕机后，M2自动切换作为写主机，当M1再上线时，便会作为读主机

分库分表

垂直拆分——分库

概念

  • 一个数据库由很多表的构成，每个表对应着不同的业务，垂直切分是指按照业务将表进行分类， 分布到不同的数据库上面，这样也就将数据或者说压力分担到不同的库上面

  • 举个例子说明：现在有两台主机，即两个mysql，也称作两个数据节点，在这两个数据节点要创建一个同名的数据库，这样方便在scheme.xml中进行分配，如果当前经过mycat有多个表的创建，此时mycat会按照配置文件中的配置信息，将需要创建的这些表分配在这两个节点上

分库原则：

  • 有紧密关联关系的表应该在一个库里，相互没有关联关系的表可以分到不同的库里。

水平拆分——分表

概念

  • 相对于垂直拆分，水平拆分不是将表做分类，而是按照某个字段的某种规则来分散到多个库之中， 每个表中包含一部分数据。
  • 简单来说，我们可以将数据的水平切分理解为是按照数据行的切分，就是将表中的某些行切分到一个数据库，而另外的某些行又切分到其他的数据库中
  • 举个例子：一个order表，里面有6条数据，其中三条分配到dn1的order表中，另外三条分配到dn2的order表中，至于按照什么方式分，有多种分片方式，后面会提

为什么要分表

  • MySQL 单表存储数据条数是有瓶颈的，单表达到 1000 万条数据就达到了瓶颈，会影响查询效率， 需要进行水平拆分（分表）进行优化。
  • 例如：例子中的 orders、orders_detail 都已经达到 600 万行数据，需要进行分表优化。

全局序列

在实现分库分表的情况下，数据库自增主键已无法保证自增主键的全局唯一。为此，Mycat 提供 了全局 sequence，并且提供了包含本地配置和数据库配置等多种实现方式

主要方式

本地文件

  • 此方式 Mycat 将 sequence 配置到文件中，当使用到 sequence 中的配置后，Mycat 会更下 classpath 中的 sequence_conf.properties 文件中 sequence 当前的值。
  • 优点：本地加载，读取速度较快
  • 缺点：抗风险能力差，Mycat 所在主机宕机后，无法读取本地文件。

数据库方式（最常用）

  • 利用数据库一个表 来进行计数累加。但是并不是每次生成序列都读写数据库，这样效率太低。
  • Mycat 会预加载一部分号段到 Mycat 的内存中，这样大部分读写序列都是在内存中完成的。 如果内存中的号段用完了 Mycat 会再向数据库要一次。
  • 如果 Mycat 崩溃了 ，那内存中的序列都没了，但是Mycat 启动后会向数据库申请新的号段，原有号段会弃用。
  • 也就是说如果 Mycat 重启，那么损失是当前的号段没用完的号码，但是不会因此出现主键重复

时间戳方式

  • 优点：配置简单
  • 缺点：18位ID过长

Mycat监控工具

是什么

Mycat-web 是 Mycat 可视化运维的管理和监控平台，弥补了 Mycat 在监控上的空白。帮 Mycat 分 担统计任务和配置管理任务。

干什么

Mycat-web 引入了 ZooKeeper 作为配置中心，可以管理多个节点。 Mycat-web 主要管理和监控 Mycat 的流量、连接、活动线程和内存等，具备 IP 白名单、邮件告警等模 块，还可以统计 SQL 并分析慢 SQL 和高频 SQL 等。为优化 SQL 提供依据。
